### What `bg-sage-terra` actually does in your compiled `main.css`

In the `themes/theme_a/static/theme_a/css/main.css` you attached, **`bg-sage-terra` is _not_ hard-coded to Sage & Stone**.

It’s compiled to:

- `background-color: hsl(var(--brand-h, 16) var(--brand-s, 46%) var(--brand-l, 43%) / …)`

So it **consumes `--brand-h / --brand-s / --brand-l`**, and only falls back to the Sage & Stone defaults **(16, 46%, 43%)** if those variables are missing.

And those fallback values correspond exactly to the classic terracotta you’re describing (`#A0563B`).

So if your CTA still looks like old Sage & Stone, you’re seeing the **fallback path**.

### That means: the problem isn’t “Tailwind hard-coding” — it’s “brand vars not applying”

Given your page source shows:

```css
:root {
  --brand-h: 161;
  --brand-s: 94%;
  --brand-l: 30%;
}
```

…then `bg-sage-terra` _should_ turn into a strong greenish colour (roughly `#059467`), not terracotta.

So if it’s staying terracotta, one of these is true:

1. **Those `--brand-*` variables are not actually applied in the browser cascade** (despite appearing in “view source”), or
2. **That CTA isn’t actually using the `.bg-sage-terra` rule at runtime** (overridden by another selector / different class than expected).

### Two 10-second checks that will tell you which it is

In browser DevTools console:

1. **Are the brand variables really applied?**

```js
getComputedStyle(document.documentElement).getPropertyValue("--brand-h");
```

- Expect: `"161"` (or whatever you set)
- If it’s empty: your inline branding `<style>` is being ignored/blocked/not in the live DOM (CSP is the classic cause).

2. **What rule is actually setting the CTA background?**

- Inspect the button → Computed → `background-color`
- If it points at `.bg-sage-terra` but still shows terracotta, then the vars are missing.
- If it points somewhere else, then you’ve got an override (template markup or a more specific CSS rule).
